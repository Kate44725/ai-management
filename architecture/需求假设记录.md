# AI算力管理系统 - 需求假设记录

> 最后更新：2026-03-05

## 一、组织架构

| 假设 | 内容 |
|------|------|
| H1.1 | 模型服务只提供给同三级部门下的其他四级部门 |
| H1.2 | 使用方开发人员可能来自一级部门下的任何层级部门（二级/三级/四级） |
| H1.3 | 组织架构：集团 → 一级部门 → 二级部门 → 三级部门 → 四级部门 |

## 二、项目与额度

| 假设 | 内容 |
|------|------|
| H2.1 | "项目"指业务项目，该项目承担AI算力费用 |
| H2.2 | 项目是动态变化的（会新建、解散） |
| H2.3 | 一个用户最多属于3个项目 |
| H2.4 | 用户额度由项目经理分配，项目经理与用户无明确汇报关系 |
| H2.5 | 不存在上层部门预算分配到项目，费用直接分摊到项目 |

## 三、用户使用场景

| 假设 | 内容 |
|------|------|
| H3.1 | 约90%用户只使用1个项目的资源 |
| H3.2 | 约10%用户使用多个项目的资源 |
| H3.3 | 期望用户能自动使用某个项目的额度，体验友好 |
| H3.4 | 项目经理需要能看到：哪个用户、哪个工具平台在使用Token |

## 四、技术栈

| 假设 | 内容 |
|------|------|
| H4.1 | IDE为自研IDE |
| H4.2 | 系统三技术栈未知，可能是多样化的 |
| H4.3 | 系统三有统一的身份认证机制 |

## 五、计费

| 假设 | 内容 |
|------|------|
| H5.1 | 按月汇总出账单 |
| H5.2 | 需要实时查看Token使用情况 |

## 六、系统一（NEW API）

| 假设 | 内容 |
|------|------|
| H6.1 | 使用开源代码部署，原则上不修改源码 |
| H6.2 | 需要支持API Key生成、Token计量 |
| H6.3 | 系统一和系统二数据库共享，系统二可直接查询Token使用量 |

## 七、API Key设计

| 假设 | 内容 |
|------|------|
| H7.1 | 方案A/B Key粒度：用户 + 项目 + 工具（即一个用户在一个项目的一个工具中有一个Key） |
| H7.2 | 方案C Key粒度：用户 + 工具（即一个用户在一个工具中有一个Key） |
| H7.3 | 方案A/B：用户不同项目的额度通过不同的Key来管理 |
| H7.4 | 方案C：用户无额度限制，Key不关联项目 |

## 八、系统三接入

| 假设 | 内容 |
|------|------|
| H8.1 | 系统三接入方式：方式A（加Key）或方式C（配置地址） |
| H8.2 | 工具类型：IDE、仿真工具、Web平台等 |
| H8.3 | 系统三有统一身份认证 |

## 九、多项目用户

| 假设 | 内容 |
|------|------|
| H9.1 | 90%用户只用1个项目，10%用户用多个项目 |
| H9.2 | 多项目用户使用模式：并行或串行都有可能 |
| H9.3 | 项目资源严格分开（涉及费用分摊） |
| H9.4 | 期望方案：默认项目 + 手动切换 |
| H9.5 | 默认项目由系统三本地保存，系统二不感知用户选择 |

## 十、系统二API设计

| 内容 | 说明 |
|------|------|
| API1 | GET /api/v1/user/projects - 获取用户项目列表+Key |
| API2 | GET /api/v1/usage/stats - 获取Token使用统计 |

## 十一、方案选择

| 方案 | 描述 |
|------|------|
| 方案A | 系统二中设置默认项目，使用时获取默认项目对应的Key |
| 方案B | 系统三中动态选择项目，使用时选择项目获取对应Key |
| 方案C | Token按项目均分结算，用户无额度限制，无需选择项目 |

## 十二、遗留问题

| 问题 | 说明 |
|------|------|
| L1 | 选择哪个方案（A/B/C）（待决策） |
| L2 | 默认项目设置：项目经理设置 vs 用户自己设置（待决策） |
| L3 | 多项目用户并行/串行使用的具体场景（待调研） |
| L4 | 系统三接入意愿和具体技术栈（待调研） |
