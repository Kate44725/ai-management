## 问题13：默认项目设置位置

### 方案A：系统二（AI算力管理系统）中设置

```
用户 → 登录系统二 → 个人中心 → 设置默认项目
系统三后端 → 查询系统二API获取用户默认项目
```

**优点：**
- 统一入口，额度管理集中
- 用户只需要在一个地方改
- 项目经理可以帮用户设置

**缺点：**
- 用户需要在系统二和系统三之间切换
- 系统三需要调用系统二API获取默认项目

**工作量：** 系统二增加"设置默认项目"功能 + 系统二提供API

---

### 方案B：系统三中动态选择项目

```
用户 → 在IDE/工具中选择项目 → 系统三后端匹配对应Key
```

**优点：**
- 用户使用体验好，在使用的地方直接选择
- 实时生效，不需要跨系统设置
- 不需要提前设置默认项目

**缺点：**
- 每次使用都需要选择（或缓存）
- 项目经理无法强制指定默认项目

**工作量：** 系统三需要实现项目选择器

---

### 推荐

**推荐方案B：** 系统三中动态选择项目

理由：
- 用户在系统三直接选择，更直观
- 系统二不感知用户选择，简化系统二逻辑
- 符合H9.4"期望方案：默认项目 + 手动切换"

## 方案C：Token按项目比例结算

### 核心特点
- Key粒度：一个用户一个工具一个Key（不关联项目）
- 用户不限制额度，按实际使用结算
- 挂靠多个项目时，按配置的比例计算至项目（如用户A挂靠项目B 70%、项目C 30%，使用1000 tokens则项目B计700，项目C计300）
- 项目经理启用/禁用用户挂靠，并配置挂靠比例

```
用户 → SuperIDE/工具调用AI → 系统根据用户+工具获取唯一Key → 调用NEW API
                                       ↓
                              系统二记录用户Token消耗
                                       ↓
                              月末按配置比例结算至项目
```

**优点：**
- 用户体验最好，无需选择项目
- 无额度限制，使用灵活
- 结算灵活，按配置比例计算

**缺点：**
- 无法事前控制费用
- 需要信任用户的使用行为

**工作量：** 系统二需实现挂靠管理和比例结算功能

## 问题17：后端如何感知当前项目

### 方案A：系统三后端直接存储用户设置

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   系统三前端     │────▶│   系统三后端     │────▶│   NEW API      │
│   (IDE/工具)    │     │   (存储设置)    │     │                │
└─────────────────┘     └─────────────────┘     └─────────────────┘
                              │
                              ▼
                       ┌─────────────────┐
                       │  用户设置表      │
                       │  user_id        │
                       │  current_project│
                       │  updated_at     │
                       └─────────────────┘
```

**优点：**
- 系统三自主可控
- 响应速度快，无需跨系统调用
- 适合系统三定制化程度高的情况

**缺点：**
- 用户在系统二修改默认项目后，系统三不会自动同步
- 每个系统三都需要独立实现
- 无法让项目经理统一管理

**工作量：**
- 系统三后端：1-2人天
- SDK：0.5人天

---

### 方案B：系统二提供API，系统三后端查询（推荐）

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   系统三前端     │────▶│   系统三后端     │────▶│   系统二API    │
│   (IDE/工具)    │     │   (查询默认项目) │     │   (返回Key)    │
└─────────────────┘     └─────────────────┘     └─────────────────┘
                              ▲                      │
                              │                      ▼
                              │              ┌─────────────────┐
                              │              │   NEW API       │
                              │              │                  │
                              └──────────────┘  (Token计量)    │
                                             └─────────────────┘
```

**优点：**
- 统一管理默认项目设置
- 项目经理可以在系统二查看/修改用户默认项目
- 用户设置后立即生效（通过API实时获取）

**缺点：**
- 增加系统间调用延迟（几十毫秒）
- 系统二API需要提供给系统三调用（需要鉴权）

**工作量：**
- 系统二：2-3人天（新增API）
- 系统三后端：1人天

---

### 推荐

**推荐方案B：** 系统二提供API统一管理

理由：
1. 统一管理default项目，项目经理可控
2. 用户设置后立即生效
3. 符合"项目资源严格分开"的要求

## 关键架构设计

### 安全性：开发者无法看到Key

```
系统三前端(IDE)  ──请求──▶  系统三后端  ──携带Key──▶  NEW API
                         │
                         │  Key存储在系统三后端
                         │  前端无法接触到Key
                         ▼
                  ┌─────────────────┐
                  │  加密Key存储     │
                  │  (AES加密)       │
                  └─────────────────┘
```

### 流程：用户调用AI

```
1. 用户在IDE点击"AI助手"
2. IDE发送请求给系统三后端
3. 系统三后端调用系统二API获取用户的默认项目+Key
4. 系统三后端使用对应Key调用NEW API
5. NEW API返回结果 + 记录Token使用量
6. 系统三后端返回结果给IDE
```
